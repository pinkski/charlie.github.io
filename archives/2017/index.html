微信小程序探究

Jan 03, 2017

Background

时间轴

  1. 2016年1月9日    微信小程序启动日
  2. 2016年1月11日    张小龙首次公开演讲透露正在探究新的公众号形态，即微信小程序
  3. 2016年11月3日夜    微信小程序正式公测
  4. 2016年11月28日    张小龙公开演讲公开微信小程序产品形态
  5. 2017年1月9日    微信小程序正式推出时间

产品形态

  1. 无消息推送
  2. 无微信入口、无应用商店
    • 扫码使用
    • LBS提示附近小程序
  3. 不能分享到朋友圈
    • 能分享到聊天或群聊
    • 分享的内容是活的，直接能与消息产生交互
  4. 不能做游戏
  5. 提供有限的搜索能力
    • 用户触到才出现，不主动推荐
  6. 和公众号相互独立

Native or Web?

初一看，觉得微信小程序是Native的，但到底是Native还是Web不好直接下定义，因为微信在这一块是保密的，下面将一层一层的摸索分析，并搜集证据证明自己的猜想。

架构

首先看看大概的架构，包括了WXML、WXSS、MINA框架，前两者分别对应了HTML、CSS，而MINA作为一个MVVM框架无论是作用还是表现都跟Vue很像。

注意到两个比较大的js文件：

  1. WAService.js
    • MINA框架代码，提供一些基础功能
  2. WAWebView.js
    • View基于它来运行

基于对这两个文件大概的分析，是不是可以猜想它依然还是运行在Webview中的，由此得到如下一个简单的架构图

  • 微信开发者工具中运行了两个页面，一个是View对应的WXML页面，另一个是Service对应的MINA框架
  • 两个页面并不互通，Service无法操作到View中的DOM和BOM，因此外部无法使用jQuery等框架
  • 开发者工具中运行：基于Chrome
  • 移动设备上的运行：Service应该是运行在X5内核(Android)和JavaScriptCore(iOS)独立的沙箱里；View还是基于Webview运行，但是为了增强体验，通过微信的平台提供一些原生的标签比如Map、Video等

小结

通过以上从IDE到代码运行的分析，可以得出一些结论

  • 目前还是Webview渲染
  • 对于开发者都是黑盒，具体的运行机制还没公开
  • 以后可能会切换成Native渲染
  • 需按照官方规定的规范开发

框架

目录结构

  1. 小程序包含一个描述整体程序的 app 和多个描述各自页面的 page
  2. app由app.js 、 app.json 、app.wxss组成，放在根目录下
  3. page由js 、 wxml 、wxss 、json组成

Service（逻辑层）

由JavaScript编写

  1. 增加了程序和页面的生命周期方法
  2. 每个页面有独立作用域和模块化能力
  3. 丰富的API，调起微信特有功能，如扫一扫、支付、微信用户数据等
  4. 脱离DOM，数据驱动，非浏览器环境，不能BOM操作，比如document、window等
  5. 代码被打包成一份JavaScript，启动时运行，结束时销毁

页面生命周期

作用域 & 模块化

  • 在JavaScript文件中声明的变量和函数只在该文件中有效，不同的文件可声明相同名字的变量和函数
  • 公共代码抽离为一个单独的js文件，作为一个模块，可用module.exports 或exports对外暴露

API

View（视图层）

由WXML和WXSS编写，由Component展示

  1. Service的数据转成View，View的事件发送给Service
  2. WXML描述页面结构
  3. WXSS描述页面样式
  4. Component是View的基本单元

小程序 VS Weex VS ReactNative

因为Weex和ReactNative的核心理念没什么差异，而Weex解决了大部分ReactNative的痛点，并且Weex引入的Vue和小程序的MINA框架类似，性能和开发成本上都占据优势，如果想要详细了解Weex和ReactNative之间对比的可以翻看我的这篇文章，下面将主要为小程序和Weex之间的对比

产品形态的异同

  1. 小程序旨在实现公众号的另一种形态，而Weex旨在实现Native生态的所有功能
  2. 小程序定位受限于微信平台，而Weex定位不受限制
  3. 小程序目前无法取代真正的Native应用，而Weex更有可能
  4. 小程序的运行依赖于微信平台，则如果给Weex提供相应的平台同样可以实现

技术实现的异同

  1. 小程序目前基于Webview渲染和部分微信平台提供的Native组件，而Weex基于JavaScriptCore解析jsbundle结合Framework生成对应的Native组件渲染
  2. 小程序提供了微信开发者工具，而Weex的开发不受IDE限制
  3. 小程序代码开发基于MINA框架受各种规范限制，而Weex基于Vue组件不受限制
  4. 小程序和Weex都能基于Chrome调试
  5. 小程序提供Native端运行的解决方案，而Weex除了Native还能在浏览器运行

性能对比

H5 <= Hybrid <= 微信小程序 <= ReactNative <= Weex <= Native

小程序 VS Google PWA

产品形态的异同

  1. 允许用户把觉得有用的应用保留下来，而不用去商店安装
  2. 零阻力、零安装、易于共享传播
  3. 不依赖网络，可离线使用
  4. 像本地应用一样交互
  5. 得益于微信平台Native的支持，小程序性能更占优势
  6. 小程序的发布有大小限制
  7. 对于一个简单的应用，开发成本上小程序更占优势；而相对复杂的比如邮箱，PWA则更有优势

技术实现的异同

  1. 小程序在移动端的表现力基于webview和部分native标签(微信平台支持)
  2. PWA离线缓存技术用到了Service Worker,它是一个更强大的AppCache
  3. PWA支持H5最新推送通知的标准
  4. 小程序对于开发者保持黑盒，PWA对于开发者是开源，所以资源和经验更占优势

坑

大坑

受限于小程序的运行环境并非一个标准的浏览器环境，微信封装工作并不完善，导致以往的大量开发经验在小程序里并不适用，几个主要的问题在于

  1. 无法调用NPM包
    • 原因：微信开发者工具不支持CommonJS标准
  2. 无法使用Babel转码
    • 结果：无法使用ES6/7的特性，callback是件麻烦事儿
  3. 无法重用组件
    • 仅支持视图模块的重用，无法抽取公共逻辑

其实，上面的坑也不是无解，通过 npm install -g labrador-cli 安装Labrador工具解决。

小坑

  1. 功能存在不少的局限性，比如导航栏颜色、map组件问题
  2. 开发者工具不支持方向键、修改样式无只能提醒、不支持全局搜索
  3. wx.request中method:”POST”模式下，后台收到请求但是data:{}数据消失，会发送一个空请求，解决方案是”Content-Type”要修改为”content-type”

Q & A

  1. 小程序为什么没有应用商店？

    • 从竞争角度，苹果的APPStore作为直接竞争对手，这是一大阻力
    • 从产品角度，小程序定位便是公众号的另一种形态
    • 从开发角度，保留用户原始习惯的基础上，节省了大部分的开发成本

  2. 小程序为什么要去中心化？

  3. 小程序能带来什么变化？

    • 让公司降低开发成本，便于商业模式试错
    • 让开发者更加注重服务的价值
    • 让用户使用更加便捷

  4. 小程序在微信里的展现模式？

    • 分享的消息可以互动，比如：分享到群聊的投票消息，其他用户不用打开消息，而是直接在消息上进行互动投票
    • 小程序可以挂起到聊天页顶部，如下图

总结

个人认为ReactNative的思想是非常适合应用到微信小程序里的，至于微信小程序为什么选择没有使用，我想应该有两方面的原因：

  1. 从技术角度考虑，和ReactNative自身的一些瓶颈有关，比如版本不稳定、listView性能问题、真机调试繁琐等
  2. 从产品形态定位考虑，小程序出发点就是微信公众号的另一种形态，结合微信平台提供的Native组件和越来越成熟的SPA技术就能带给用户比较好的体验

由此可以预见，如果上述原因中技术问题得以解决，那么产品形态的定位便可以大大扩展，而Weex做的事情中解决了大部分ReactNative的瓶颈问题，那么使用Weex结合微信小程序的思想所得到的产品形态势必可以大大的扩展。

前端性能优化基础到进阶

Jan 01, 2017

学习前端性能优化需要会什么？

说道前端性能优化，可能你第一个想到的是yahoo35条军规这种通用方案，因为它的确已经总结得非常全面，但在实际项目场景下yahoo35条军规是否适用?除了yahoo35条军规还有哪些可以达到性能优化的手段？这些都是需要结合实际的项目去实践而并非生搬硬套的使用yahoo35条军规

当然，想要灵活运行技术手段进行前端性能优化必须要知道一些基础知识，比如在浏览器地址栏输入一个URL，到浏览器为我们展示出页面，这其中的过程发生了什么？

前端性能优化是什么？

yahoo35条军规

yahoo35条军规详细阐述

当然，yahoo35条军规只是通用的优化方案，它也有不适合的场景，比如一个网页游戏页面，优先加载的肯定是图片和背景音乐

网络优化

IP优化策略

如果网络大多基于TCP连接，为了将DNS时间降至最低，可以采用动态IP优化策略算法，即内置Server IP列表，该列表可以在App启动服务中下发更新。App启动后首次网络服务从Server IP列表中取一个IP地址进行TCP连接，同时DNS解析会并行进行，DNS成功后，会返回最适合用户网络的Server IP，这个Server IP会被加入到Server IP列表中被优先使用。

通信数据格式优化

数据传送格式转换到Protocol Buffer，并对数据进行Gzip压缩，使用 PB+Gzip 数据大小下降约80%，另外FlatBuffer(Protocol Buffer的一种进化方案与实现)在性能上可能获取更大的优势。

弱网和网络抖动优化

根据不同的网络质量改变网络服务策略：

  1. 调整长连接池个数：例如在2G/2.5G Egde网络下，会减少长连接池个数为1；Wi-Fi网络下可以增长长连接池个数等机制。
  2. 动态调整TCP Connection、Write、Read的超时时间。
  3. 不同网络类型状态切换连接机制
其它网络优化
  1. 在应用层面引入重试机制提高网络服务成功率
  2. 网络服务优先级机制，高优先级服务优先使用长连接，低优先级服务默认使用短连接
  3. 网络服务依赖机制，根据依赖关系自动发起或取消网络服务，例如主机服务失败时，子服务自动取消

复用（维护性、开发成本）

加载方式（提升用户体验）

前端性能优化怎么用？

Hybrid框架优化

  1. 离线组件包

    不同于HTML5，组件化Hybrid方案引入了组件包的概念，将业务需要的所有静态资源打包成一个组件包文件，并进行离线化版本管理，而且关于离线包的更新是采取增量差分升级更新的方式

    用户访问组件页面之前，组件默默将新业务组件包下载至客户端，用户访问组件页面时，WebView加载的实际上均是本地页面和资源，大幅提升页面加载性能。不仅如此，关键业务组件在APP发布时会预置到App中，减少首次下载安装的时间。

  2. WebView 预加载

    影响页面加载速度的因素非常多，我们在对 WebView 加载一个网页的过程进行调试发现，每次加载的过程中都会有较多的网络请求。除了 Web 页面自身的 URL 请求，还会有 Web 页面外部引用的 JS、CSS、字体、图片等都是个独立的 HTTP 请求。这些请求都是串行的，加上浏览器的解析、渲染时间就会导致 WebView 整体加载时间变长，消耗的流量也对应地增多。所以为了加快 WebView 的整体加载时间，我们使用了预加载策略，将 JS、CSS、字体、图片等资源提前加载出来。

    Android 通过 WebView 自带的 API 即 setWebViewClient 中的回调接口 ShouldInterceptRequest 去实现资源的预加载。

    iOS 通过 setNeedsLayout 机制提前初始化下一个 UIwebView，将 HTML 文件初始化，然后捕获点击事件去动态刷新数据。

图片优化

地图优化

  1. 数据分段加载
  2. 数据缓存机制
  3. WebView单独WebCore的优化
  4. 提前预加载数据和JS资源等优化

启动内存优化

  1. 图片占用内存
  2. 轻量级数据结构
  3. 内存泄露

ReactNative优化

  1. ListView没有实现cell重用，导致数据量大的时候卡顿

    • 借鉴ReactNativeTableView的实现，开发可重用cell的ListView，接口和官方原生保持一致
    • Android借鉴iOS方案，采用RecyclerView实现
    • 常用的下拉刷新，载入更多，右侧字母栏索引加入

  2. RN加载最大的瓶颈 JS Init + Require，也就是jsbundle执行时间

    • 将框架代码拆分，让所有业务使用一份框架代码，拆分为Common.js 和 Business.js
    • ReactNative的框架部分文件(Common.js)大小比较大，占用了绝大部分的JS执行时间，把这部分的执行加载时间放到后台预先做完，进入业务只需要执行页面页面的几个JS
    • 设计一个空白页面的Fake App，用来监听要显示的真实业务JS模块，收到监听后，渲染业务模块最终显示页面
